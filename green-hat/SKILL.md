# Green Hat — Alternatives & New Ideas

The Green Hat is generative. Its job is not to judge the current idea but
to expand the solution space. What else could achieve the same goals? What
hasn't been considered? What if the problem itself is framed wrong?

The Green Hat is the most uncomfortable hat for analytical thinkers
because it requires suspending judgment temporarily. The other hats
evaluate. This hat creates.

## Process

### Step 1: Restate the underlying problem

Before generating alternatives, clarify what problem the original idea is
actually trying to solve. Often the document conflates the solution with
the problem. Separate them:

- What is the core problem or opportunity?
- Who has this problem? How badly?
- What would "solved" look like from the user/customer perspective?
- Is the document solving the right problem, or a proxy problem that's
  easier to articulate?

Sometimes the most valuable Green Hat output is realising the document
is solving the wrong problem entirely.

### Step 2: Generate alternative approaches

For each major element of the proposal, ask: what else could work?

**Category 1: Different solutions to the same problem**
- How would you solve this if the proposed approach didn't exist?
- How would a well-funded competitor solve this?
- How would a scrappy startup with 1/10th the resources solve this?
- How was this problem solved before the current approach was proposed?
- What would the lazy/simple/boring solution look like?

**Category 2: Different scope**
- What's the 10× bigger version of this idea?
- What's the stripped-down version that delivers 80% of the value?
- What if you only solved this for one segment of the audience?

**Category 3: Different sequence**
- What if you did step 3 first instead of step 1?
- What if you validated the riskiest assumption before building anything?
- What if you shipped the smallest possible thing and learned from it?

**Category 4: Different framing**
- What if the problem statement is wrong?
- What if the constraint everyone accepts isn't actually a constraint?
- What would someone from a completely different industry do?

### Step 3: Design minimum viable tests

For both the original idea and the most promising alternatives, propose
the smallest experiment that would test the core hypothesis:

- What is the riskiest assumption? (The one that, if wrong, makes
  everything else irrelevant)
- How could you test that assumption in days or weeks, not months?
- What would you measure, and what result would make you proceed vs.
  stop?
- Can you test with existing data, existing customers, or a manual
  process before building anything?

### Step 4: Combine and remix

Look for hybrid approaches:

- Can the strongest parts of the original idea be combined with a
  different execution approach?
- Can one of the alternatives serve as a stepping stone to the
  original vision?
- Is there a sequence that de-risks the original idea — do the cheap
  safe thing first, graduate to the ambitious thing once it's validated?

### Step 5: The uncomfortable question

Propose at least one alternative that challenges a sacred cow — an
assumption that the document treats as given but could be questioned:

- What if we didn't do this at all? What would happen?
- What if the opposite approach is correct?
- What if the biggest risk isn't failure but success at the wrong thing?

## Output format

```markdown
## Green Hat Analysis — Alternatives & Ideas

### The real problem
[Restated core problem, separated from the proposed solution.
Note if the framing itself might be wrong.]

### Alternative approaches

#### 1. [Name of alternative]
- **Approach:** [What you'd do differently]
- **Advantage over original:** [Why this might be better]
- **Risk:** [Why this might not work]

#### 2. [Name of alternative]
...

#### 3. [Name of alternative]
...

### Reframing
[Is there a different way to define the problem that opens up
better solutions? Challenge a foundational assumption.]

### Minimum viable tests
For the original idea:
- **Riskiest assumption:** [what to test]
- **Test:** [how to test it cheaply]
- **Decision criteria:** [what result means go vs. stop]

For the most promising alternative:
- **Test:** [how to validate it]
- **Decision criteria:** [what you'd need to see]

### The uncomfortable question
[The sacred cow challenge — what if we didn't do this at all,
or what if we did the opposite?]
```

## Tone

Inventive, lateral, unencumbered. The Green Hat is a creative director
who's read the brief but isn't bound by it. It respects the constraints
but isn't afraid to ask whether they're real constraints or just habits.

Avoid generating alternatives for their own sake — each should be a
genuinely plausible path that a reasonable person could pursue, not
brainstorming filler. Quality over quantity. Three strong alternatives
beat ten weak ones.

## Standalone vs. sequenced use

**Standalone deep-dive:** Go deep on all sections. Generate 4–6
alternatives with real depth. This mode is valuable when the user is
stuck on an approach and wants to see what else is possible, or when
a team is locked into groupthink and needs an outside perspective.

**As part of six-hat sequence:** 2–3 alternatives, one reframing, and
the minimum viable test for the original idea. Keep it to roughly one
page — the Blue Hat closing will synthesise.
